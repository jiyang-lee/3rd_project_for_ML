1왜 관리자가 돈 주고 써야 하나

관리자가 하루 종일 계기판, 민원 엑셀, 규정, 과거 작업 이력을 뒤지는 시간을 줄이고, 위험 단지와 점검 순서를 카드로 받기 때문입니다. 핵심 비용 절감은 조치 지연, 불필요한 출동, 보고서 작성, 책임구간 확인에서 발생합니다.

  • 위험 단지/기계실 랭킹으로 오늘의 순서가 바로 보인다.
  • 모델 점수와 근거가 같이 있어 왜 1순위인지 설명 가능하다.
  • 보고서, 업체 전달문, 입주민 안내문을 즉시 초안화한다.

2LLM 비용을 어떻게 줄이나

원시 센서 시계열 전체를 LLM에 넣지 않습니다. 코드와 모델이 먼저 점수, 근거, review flag를 만들고 LLM은 Top-N 카드와 선택된 근거만 받아 설명과 문서를 만듭니다.

  • 일일 배치에서는 Top 10-30개 카드만 LLM 후보로 전달한다.
  • 규정/RAG와 작업 이력 조회는 버튼 클릭 또는 질문 때만 호출한다.
  • 같은 카드의 보고서와 문구는 캐시하고, 템플릿을 우선 사용한다.

3왜 LLM이 필요한가

ML/DL은 위험 점수와 라벨을 만들 수 있지만 “관리자가 어떤 순서로 누구에게 무엇을 말해야 하는지”까지 안정적으로 문장화하지 못합니다. 책임구간, 규정, 민원 문장, 보고 톤은 언어와 문맥 판단이 필요합니다.

  • 모델 점수를 사람 말로 압축하고 근거를 연결한다.
  • 규정, SOP, 민원, 작업 이력을 한 질문 안에서 조합한다.
  • 업체 전달문, 내부 보고, 입주민 공지를 대상별 톤으로 바꾼다.

4필요한 데이터/DB/Tool은 무엇인가

현재 ZIP은 모델 score와 agent card까지는 준비되어 있습니다. 실제 서비스가 되려면 민원, 작업 이력, 규정/SOP, 책임구간, 업체 연락처를 운영 DB와 RAG Tool로 붙여야 합니다.

  • 필수 DB: sensor windows, model scores, agent cards, complaints, maintenance history, document chunks.
  • 필수 Tool: SQL 조회, 유사사례 검색, 규정 RAG, 책임구간 라우터, 보고서/전달문 생성.
  • 필수 로그: LLM 입력/출력, 사용 토큰, 근거 카드 ID, 사람이 수정한 최종 문장.

LLM 호출을 줄이는 경계 설계

비싼 LLM은 마지막 20%에만 사용합니다. 반복 계산, 랭킹, threshold, score 결합은 코드와 모델이 담당합니다.

1. 모델 배치센서 window 전체를 코드로 점수화. LLM 호출 없음.
2. 카드 압축priority, why_reason, recommended_action 필드 생성. LLM 호출 없음.
3. Top-N 선별urgent/high 또는 review_required만 후보로 전달.
4. 도구 조회선택된 카드 ID로 규정, 이력, 민원만 부분 조회.
5. 문장 생성관리자 질문 또는 버튼 클릭 시에만 LLM 실행.
금지: raw CSV 전체 prompt 투입 권장: card_id 기반 조회 권장: 템플릿 우선 권장: 결과 캐시 권장: token/cost log
